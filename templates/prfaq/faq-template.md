# FAQ Template

> Write the FAQ from the customer's perspective first, then internal. Every question should be one a real person would ask — not a setup for marketing copy.

---

## Customer FAQ

### Q: [MOST OBVIOUS OBJECTION — "Why would I use this instead of X?"]
A: [Honest answer. Acknowledge the alternative. Explain the specific gap this fills.]

### Q: [SECOND MOST COMMON QUESTION — usually about data/privacy/security]
A: [Specific technical answer. Mention encryption, data residency, compliance if relevant.]

### Q: [PRICING QUESTION — "How does billing work?"]
A: [Concrete numbers. Per-unit breakdown. What happens if usage spikes.]

### Q: [INTEGRATION QUESTION — "Does it work with my existing tools?"]
A: [List specific integrations. Be honest about what's not supported yet.]

### Q: [RELIABILITY QUESTION — "What happens if it goes down?"]
A: [SLA, failover, data persistence. Specific numbers.]

### Q: [EXIT QUESTION — "Can I leave if I want to?"]
A: [Data portability. Export format. Lock-in or lack thereof.]

### Q: [UNIQUE-TO-YOUR-PRODUCT QUESTION]
A: [Answer.]

---

## Internal FAQ

### Q: [HARDEST TECHNICAL QUESTION — the thing engineering will ask]
A: [Honest answer about what's hard and how you'll solve it.]

### Q: [BUSINESS MODEL QUESTION — "How do we make money?"]
A: [Unit economics. CAC, LTV, margin if known.]

### Q: [COMPETITIVE RESPONSE QUESTION — "What if [COMPETITOR] copies this?"]
A: [Moat. Speed. Distribution advantage. Be specific.]

### Q: [SCALING QUESTION — "What breaks at 10x / 100x?"]
A: [Architecture limits. What needs to change. When.]

---

## Quality Checklist

- [ ] Every customer question is one a real person would type into a support chat
- [ ] No question exists solely to set up a marketing answer
- [ ] Internal questions include the uncomfortable ones
- [ ] Answers include specific numbers where possible
- [ ] The exit question is answered honestly
